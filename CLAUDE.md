# 大类资产看板 —— 项目长期工作准则

本文件是这个项目**默认、长期**的工作要求,适用于以后所有改动(黄金页及后续美债/美股等页)。
与全局 `~/.claude/CLAUDE.md` 互补;优先级:用户当下显式请求 > 本文件 > 全局默认。
"包括但不限于"——下面是已沉淀的要求,新的约定随时补进来。

## 一、核心理念:奥卡姆剃刀 + 科学性(并重,缺一不可)

- **奥卡姆**:只做有增量价值的改动。不过早抽象;不加用户没要的可配置/扩展/防御性处理;更短且同样清晰的实现优先;不做"顺手"重构。
- **科学性**:每个指标、口径、计算方法都要**可溯源、可对账、可证伪**。引入的统计方法要经得起推敲。复杂度按需引入;本轮搁置的方法论局限要**显式留备注**(如 spec 的"已知局限"节),绝不静默跳过。
- 两者冲突时不要二选一:用最简单的、但站得住脚的方案。明显过度复杂的方案要反对它。

## 二、永远从三个维度想,不只前端

1. **观察的指标**:看的是不是对的东西;是否有增量信号;是否与已有指标共线、重复计数。
2. **观察的方式**:状态/评分/相关性的计算口径是否站得住;窗口、阈值、显著性是否诚实。
3. **数据获取**:来源、频率、披露滞后、可复现性;优先自动化,但尊重环境约束。

提改进时,把这三轴和"不一定有问题、只是指出可能性"分开说清楚,标注把握程度。

## 三、数据与诚实(本项目的红线)

- **先对账再切换**:换数据源前,与旧源在重叠区间对账证明一致;不一致就 fail-closed,不绕过、不用 `--no-reconcile` 之类强过。
- **staleness/缺失显式化**:不用硬编码 `fresh`;按频率算真实滞后并显示;缺数据标 `missing`,**绝不用假数据或旧值掩盖**。月末未来日期等边界要处理(滞后不为负)。
- **不制造过度精确感**:弱信号(如 EPU/地缘)可纳入评分,但 UI 文案克制(代理 ≠ 因果驱动),并用关系检验卡暴露其**真实有效性**。看板越漂亮,越要把数据质量和 caveat 放在视觉显眼处。
- **防自证循环**:用派生自标的自身的信号做关系检验时,必须用**未来收益(前瞻)**,不能用同期收益,否则是同义反复。
- 保留 `Wrong If` / 失效条件,防止判断变成口号。

## 四、流程与工程纪律

- **先讨论再做**:涉及指标/方法/数据口径/破坏性/用户可见行为的改动,先指出可能性、与我对齐,再动手。
- **增量、可逆、可验证**:优先小步;大迁移分阶段推进(如 Excel → Wind 逐步退役),每步可单独验证。
- **测行为不测实现**:断言要有判别力,别写恒真的烟测;**别把易变的日期/数值钉死**(改测来源/区间/质量状态)。永不为过检而弱化、跳过、删除测试。
- **反馈批判式吸收**:对评审/反馈先验证再采纳;该用技术理由反驳就反驳,不做表演式同意;反驳错了就如实更正。
- **复用现有栈**:`unittest` + 标准库 + `openpyxl`;数据走 `tkf_wind`/FRED/CFTC;不擅自引入新依赖。
- **提交规范**:commit/PR 用英文 conventional 风格,结尾加 `Co-Authored-By` trailer;特性走分支;`git push`、发布、部署等外向操作先确认。

## 五、本项目运维约束(易忘)

- **Wind 是 LAN-only**(`10.92.26.150`),GitHub CI 到不了。更新数据必须本机跑 `scripts/refresh_wind_data.py` + `scripts/refresh_market_data.py` 并提交 CSV;CI(`pages.yml`)只渲染已提交的数据。
- **部署**:push 到 `main` 触发 GitHub Pages 重新部署 `site/index.html`。要数据新,先本地刷新再 push。
- 设计依据沉淀在 `docs/superpowers/specs/` 与 `docs/superpowers/plans/`;下一轮 backlog 见 spec 的"已知局限"节。
